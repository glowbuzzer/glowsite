## Introduction

## Tasks

Within &gbc, Tasks are an assembly of Activities that are executed in sequence in response to an event occurring.

&gbc can have multiple Tasks defined and tasks can contain multiple Activities.

&gbc will execute a Task based on its current state (E.g. whether it is already running or has finished) and whether the task's Trigger has fired.

The trigger for a task would typically be an event like a IO being set or tasks can be triggered from front-end code.

Tasks allow for JavaScript code to be run is response to events occurring or periodically without using the front-end code (browser based). See <Link to="/docs/gbc/response_times"> here </Link> for more details on the response time of code in different layers of GBC.

Tasks can have trigger to start them running. This could be a digital input or a thresholded analog input.

## Activities

### Introduction

Within &gbc, an Activity is a machine control primitive. They generally do one of two things:

* Read and write to the machine's IO. E.g. set a Digital Out or read an Analogue Input)
* Move joints (motors) either individually or as a set in concert. E.g. rotate a motor at 50rpm or move a robot along a path)

Activities can be invoked in three different ways:

1. As a`solo activity`
   - this is invoked by sending a websockets message from the front-end control JavaScript to &gbc
   - all the activities are given a solo activity "wrappper"
   - essentially they become commands that can invoked through the websockets interface
   - you can have banks of solo activities these are not queued
   - solo activities map to a single kinematics configuration (kc)
   - when you issue a new solo activity (through the websockets interface) the current solo activity gets cancelled - these work under a "one at a time" model
   - in the React front end API, solo activities are implemented as a promise - i.e. you await a solo activity's completion
2. As a `streamed activity`
    - these are used for continuous motion. E.g. streaming a gcode program
   - they support blending - pause and resume
   - classic queue for flow control
   - capacity to check so don't overfill - front end code handles this for you.
   - Send end program activity to kick it off if you dfont fill buffer  if you fill buffer will kick off acutomaically
   - gcode reference impl for this - not exclusively
3. From a Task executing on &gbc that either cyclically or in response to an event executes the Activity


### Sending activities to &gbc

`SoloActivity`
```tsx
soloActivity.moveJoints([1000,0,0]).promise()
```

`Websockets message`

```tsx
const connection = useConnect()
        moveLine(pos: Vector3) {
            const currentTag = tag.current++
            return new Promise<void>((resolve, reject) => {
                // store the functions for later resolution/rejection
                promisesRef.current[currentTag] = { resolve, reject }
                connection.send(
                    JSON.stringify({
                        command: {
                            soloActivity: {
                                0: {
                                    command: {
                                        activityType: 2,
                                        tag: currentTag,
                                        moveLine: {
                                            line: {
                                                position: pos
                                            }
                                        }
                                    }
                                }
                            }
                        }
                    })
                )
            })

```
